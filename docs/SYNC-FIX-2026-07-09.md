# Sync fix — 2026-07-09

Журнал разбора расхождения: рабочее устройство (A) показывало 91 пациента,
второе устройство (B, v1.3.1) после синхронизации — 81.

## Бэкапы

- `~/backups/dental-patients/data-cloud-2026-07-09-91patients.json` — снимок облака
  на момент разбора: 91 пациент, `updatedAt: 2026-07-08T14:56:19.185Z`.
  Это единственный полный слепок базы. Восстановление — PUT этого файла в
  `Яндекс.Диск:/dental-patients/data.json`.

## Диагноз

Облако было в порядке: 91 пациент, все реальные, дублей `id` нет, демо-записей нет.
`updatedAt` облака совпадал с `createdAt` последнего пациента — значит A пушил исправно.

Недостающие на B 10 записей — ровно 10 последних по `createdAt` (7–8 июля).
B держал снапшот по состоянию на ~6 июля и **считал себя синхронизированным**.

Механика залипания: `pull()` сравнивает `updatedAt` целого снапшота. Если B получает
по HTTP устаревший ответ, его локальный `updatedAt` равен `remoteAt` этого ответа →
ветка `else` → `setState('idle')`. Данные не меняются, push не запускается
(`localAt` не больше `remoteAt`), UI рапортует успех. Состояние устойчивое.

## Найденные дефекты

1. **GET без анти-кэш заголовков** (`syncManager.ts`, `webdav()`).
   На Android `fetch` идёт через OkHttp с HTTP-кэшем. Яндекс.Диск на GET отдаёт
   `ETag`/`Last-Modified` без `Cache-Control`, клиент вправе применить эвристическую
   свежесть и вернуть старый снапшот. Основная гипотеза по симптому B.

2. **`loadData()` выдавал снапшоту без метки текущее время** (`storage/index.ts`).
   `updatedAt: parsed.updatedAt ?? new Date().toISOString()` — такой снапшот на каждом
   запуске получал `now()`, всегда выигрывал LWW и затирал облако.

3. **`schedulePush()` делал слепой PUT** мимо demo-защиты (`syncManager.ts`).
   `pull()` умеет `hasRealData()`, а `schedulePush` — нет. Сценарий потери: B стартует
   офлайн (pull → `offline`), врач трогает любую запись → `persist()` заливает
   демо-снапшот поверх 91 боевого пациента.

4. **`runExclusive()` тихо выбрасывал операцию** (`if (running) return;`), без ретрая.
   Push, совпавший по времени с pull, терялся до следующего запуска приложения.

5. **(НЕ ИСПРАВЛЕНО)** LWW по всему снапшоту, без слияния по сущностям. Два устройства,
   правившие разных пациентов, теряют правки одного из них. Требует тумбстоунов для
   удалений, иначе удалённый на A пациент воскреснет с B. Отдельная задача.

## Изменения (пункты 1–4)

`src/sync/syncManager.ts`
- `webdav()`: на GET добавлены `Cache-Control: no-cache, no-store`, `Pragma: no-cache`,
  `cache: 'no-store'` и cache-buster `?ts=<epoch>` в URL (кэш OkHttp ключуется по URL,
  одних заголовков мало). PUT/MKCOL не тронуты. Пятый параметр `cacheBust` позволяет
  запросить чистый URL.
- `pull()`: 404 больше не означает сразу «снапшота нет». Сначала повторный GET **без**
  cache-buster. Если сервер отдаёт файл по чистому URL — работаем с ним. Иначе, если и
  там 404 — push локального. Страховка на случай, если Яндекс.Диск сочтёт `?ts=` частью
  имени файла: без неё устройство B залило бы свои 81 поверх 91 через 404-ветку.
- `schedulePush()`: вместо `doPush(data, auth)` вызывает `pull(data)` — тот отдаёт push
  только когда локальное реально новее, либо облако демо/пусто.
- `runExclusive()`: слот `pending` вместо тихого дропа; конкурирующая операция
  вытесняет предыдущую отложенную (актуален только последний снапшот) и выполняется
  после текущей.

`src/storage/index.ts`
- `loadData()`: `?? new Date(0).toISOString()` вместо `new Date().toISOString()`.
  От затирания пустым удалённым снапшотом защищает `hasRealData()` в `pull()`.

## Trade-off, который приняли

`schedulePush` → `pull` означает, что push проходит только при `localAt > remoteAt`.
Если системные часы устройства отстают от времени последней записи в облаке, его
изменения не зальются, а локальные данные будут перезаписаны облаком. Раньше слепой
PUT заливал в любом случае. Обмен корректности на устойчивость к затиранию — сознательный.
Если у врача сбиты часы, симптом будет «мои правки не уезжают».
